## DeepSeek-V1 (DeepSeek LLM): The Foundation

The first DeepSeek release (Jan 2024), "DeepSeek LLM: Scaling Open-Source Language Models with Longtermism" — **7B and 67B dense models** trained on 2T tokens. This is the *pre-MoE, pre-MLA* baseline; its contributions are about **scaling discipline and a solid dense recipe**, not the architectural innovations that come in V2.

### Dense Transformer (NOT MoE)
- **Standard dense architecture, LLaMA-lineage.** DeepSeek-V1 is a conventional dense Transformer (every parameter active for every token). It follows the LLaMA design: **Pre-Norm with RMSNorm, SwiGLU FFN, Rotary Position Embeddings (RoPE)**. The 7B has 30 layers; the 67B has 95 layers and uses **Grouped-Query Attention (GQA)** to cut inference cost.
- **Important:** MoE is **not** part of V1. The "Top-K router / experts" idea was introduced in a *separate* contemporaneous paper, **DeepSeekMoE (16B)**, and only became part of the main line in **V2**. (Your draft attributing Top-K MoE to V1 mixes the two papers — keep them distinct: V1 = dense LLM; DeepSeekMoE = the separate MoE architecture paper.)

### Scaling Laws (the real headline of V1)
- The paper's central contribution is an empirical **scaling-laws study** resolving prior disagreements about how to trade off model size vs. data given a compute budget. These findings guided the 7B/67B configurations and data sizing — "Longtermism" in the title refers to using scaling laws to plan model growth rather than guess.
- Result: **DeepSeek LLM 67B surpasses LLaMA-2 70B**, especially on code, math, and reasoning, and the 67B-Chat beats GPT-3.5 on open-ended evaluation.

### Alignment Pipeline
- **SFT + DPO.** Base models are aligned via **Supervised Fine-Tuning** then **Direct Preference Optimization** (note: DPO, not RL/GRPO — that comes later), producing the DeepSeek Chat models.

### Domain Specialization (separate sibling models)
- DeepSeek also shipped domain models — **DeepSeek-Coder** and **DeepSeek-Math** — refined on code/logic-heavy corpora (and, for math, with techniques that later fed into the RL line). These prioritize structural/logical correctness over open-ended fluency.
- **Caveat:** these are *sibling specialist models* in the early DeepSeek family, not "checkpoints of V1" per se — they're trained separately, though they share the lineage.

## DeepSeek-V2: The Efficiency Breakthrough
* **MLA (Multi-Head Latent Attention):** MLA uses low-rank joint compression to store Key-Value information in a compact latent vector, compressing KV cache. MLA fixes this by using a "compress-then-decompress" strategy based on Low-Rank Adaptation. It reduces the memory footprint of long-context conversations by 93%, enabling context windows on 128K to 1M+ tokens.
  - Latent Vector Compression: Instead of storing large embedding for separate rows in Key and Value cache, MLA compresses them into a smaller embedding via a linear projection into a lower-dimensional space using a learned matrix of smaller rank.
  - Up-Projection: During the actual "thinking" phase (the attention calculation), the model uses a learned matrix to "up-project" or inflate that tiny latent vector back into the full Key and Value matrices needed for math.
  - Catch: Positional information (knowing that word 1 comes before word 2) is hard to compress. If we compress it, the "coordinates" get blurry. The fix is to use decoupled Rotary Positional Embeddings (RoPE): To keep track of word order, MLA separates the positional information from the content information. It only applies "rotation" (RoPE) to a small portion of the vector, which further saves computation. So we keep the contenct vector compressed, and a position vector, that are combined during decompression.
  - In technical terms, it's better than Multi-Query Attention and Multi-Head attention. MLA provides the efficiency of MQA with the complex logic of Multi-Head Attention. In standard MQA, we have one Key and one Value head that all Query heads must share. In MLA, instead of a single "Head," you store a single latent vector (the compression vector) for each token position. The "efficiency" of MQA comes from reducing Memory Bandwidth Bottlenecks, but maintains the intelligence and precision of Multi-Head Attention (where every head has its own specialized focus).
  - Also increases runtime because bandwidth is the efficiency bottleneck in inference, and this reduces the time moving memory, despite increasing the computation time on compression/decompression.

- **Fine-Grained Expert Segmentation:** An evolution of the MoE architecture. Instead of a few large experts (8 experts and top=k=1 or 2), V2 splits them into hundreds of "micro-experts" (180 experts, top-k=16). It also utilizes "Shared Experts" (cloned) that handle universal language patterns. This increases expert picking precision, i.e. it allows for hyper-specialization. Micro-experts focus on niche knowledge (e.g., specific API documentation), while the Shared Expert prevents redundant learning of basic grammar.

### MLA Workflow — Tensor Shape Trace for an Input $x$ of size $B \times S \times E$

Convention: projections written as $xW$ (input in rows, the implementation convention), so shapes read left to right.

#### Config (DeepSeek-V2)

| Symbol | Meaning | Value |
|---|---|---|
| $E = d$ | embedding / hidden dim | 5120 |
| $n_h$ | number of heads | 128 |
| $d_h$ | content dim per head | 128 |
| $d_h^R$ | RoPE dim per head | 64 |
| $d_c$ | KV latent dim (cached) | 512 |
| $d_c'$ | query latent dim | 1536 |

**Input:** $x : B \times S \times 5120$

#### Query path

1. **Down-project queries:** $c^Q = x\\,W^{DQ}$
   $W^{DQ}: 5120 \times 1536 \Rightarrow c^Q : B \times S \times 1536$
2. **Up-project to content queries:** $q^C = c^Q\\,W^{UQ}$
   $W^{UQ}: 1536 \times (128{\cdot}128) \Rightarrow B \times S \times 16384 \rightarrow$ reshape $B \times S \times 128 \times 128$
3. **RoPE queries:** $q^R = \mathrm{RoPE}(c^Q\\,W^{QR})$
   $W^{QR}: 1536 \times (128{\cdot}64) \Rightarrow B \times S \times 128 \times 64$
4. **Concat per head:** $q = [\\,q^C ; q^R\\,] \Rightarrow B \times S \times 128 \times 192$

#### Key / Value path

5. **Down-project KV (the bottleneck):** $c^{KV} = x\\,W^{DKV}$
   $W^{DKV}: 5120 \times 512 \Rightarrow c^{KV} : B \times S \times 512$ &nbsp; ← **cached**
6. **Up-project keys:** $k^C = c^{KV}\\,W^{UK}$
   $W^{UK}: 512 \times 16384 \Rightarrow B \times S \times 128 \times 128$
7. **Up-project values:** $v^C = c^{KV}\\,W^{UV} \Rightarrow B \times S \times 128 \times 128$
8. **Shared RoPE key:** $k^R = \mathrm{RoPE}(x\\,W^{KR})$
   $W^{KR}: 5120 \times 64 \Rightarrow k^R : B \times S \times 64$ &nbsp; ← **cached**; one per token, no head axis (broadcast to all 128 heads)
9. **Concat per-head key:** $k = [\\,k^C ; \underbrace{k^R}_{\text{broadcast}}\\,] \Rightarrow B \times S \times 128 \times 192$

#### Attention

10. **Scores:** $q\\,k^\top$ over the 192-dim, scaled by $1/\sqrt{d_h + d_h^R} = 1/\sqrt{192}$
    $\Rightarrow B \times 128 \times S \times S$, apply causal mask, softmax over keys
11. **Weighted values:** $\mathrm{scores}\cdot v^C \Rightarrow B \times S \times 128 \times 128 \rightarrow$ reshape $B \times S \times 16384$
12. **Output projection:** $u = o\\,W^{O}$
    $W^{O}: 16384 \times 5120 \Rightarrow u : B \times S \times 5120$

Output is back to $B \times S \times E$ — same shape as input, ready for the residual add.

#### What lives in the cache

Per token, only steps 5 and 8 are stored:

$$\underbrace{c^{KV}}_{512} + \underbrace{k^R}_{64} = 576 \text{ scalars/token}$$

KV cache shape: $B \times S \times 576$, **independent of $n_h$**. Vanilla MHA would cache $2\\,n_h\\,d_h = 2 \cdot 128 \cdot 128 = 32768$ scalars/token — a ~57× reduction.

#### Absorbed Decode Trace

During autoregressive decode, $S = 1$ (one new query token) attending against the cached history of length $S_{\text{kv}}$. The "absorb" trick folds the up-projections into neighboring matrices so the full per-head $k^C/v^C$ are **never materialized** — attention runs directly in the compressed $d_c = 512$ space.

##### Pre-compute (offline, fold weights once)

- **Fold $W^{UK}$ into the query side:** define $\tilde{W}^{Q} = W^{UQ}\\,(W^{UK})^\top$ so the content score becomes
  $q^C (k^C)^\top = (c^Q \tilde W^{Q})\\,(c^{KV})^\top$ — i.e. the query is mapped straight into the 512-dim latent space, no $k^C$ needed.
- **Fold $W^{UV}$ into the output:** define $\tilde{W}^{O} = (W^{UV})\\,W^{O}$ so the value up-projection is deferred until after the attention weighting.

##### Per decode step (new token, $S = 1$)

| Step | Op | Shape |
|---|---|---|
| D1 | Down-proj query: $c^Q = x\\,W^{DQ}$ | $B \times 1 \times 1536$ |
| D2 | Map query into latent space: $q^{\text{lat}} = c^Q \tilde W^{Q}$ | $B \times 1 \times 128 \times 512$ |
| D3 | RoPE query: $q^R = \mathrm{RoPE}(c^Q W^{QR})$ | $B \times 1 \times 128 \times 64$ |
| D4 | New token's latent + RoPE key: $c^{KV} = x W^{DKV}$, $k^R = \mathrm{RoPE}(x W^{KR})$ | $B \times 1 \times 512$, $B \times 1 \times 64$ |
| D5 | **Append D4 to cache** → history | $c^{KV}_{\text{cache}}: B \times S_{kv} \times 512$, $k^R_{\text{cache}}: B \times S_{kv} \times 64$ |
| D6 | Content scores (in latent space): $q^{\text{lat}} (c^{KV}_{\text{cache}})^\top$ | $B \times 128 \times 1 \times S_{kv}$ |
| D7 | RoPE scores: $q^R (k^R_{\text{cache}})^\top$ (key broadcast over heads) | $B \times 128 \times 1 \times S_{kv}$ |
| D8 | Sum scores, scale by $1/\sqrt{192}$, causal mask, softmax | $B \times 128 \times 1 \times S_{kv}$ |
| D9 | Weight the **latent** directly: $\mathrm{scores}\cdot c^{KV}_{\text{cache}}$ | $B \times 128 \times 1 \times 512$ |
| D10 | Apply absorbed output proj: $(\cdot)\\,\tilde W^{O}$ | $B \times 1 \times 5120$ |

##### Why this is the win

- Steps D6 and D9 operate on the **512-dim latent**, not the $128 \times 128 = 16384$-dim per-head K/V. The up-projection matrices $W^{UK}, W^{UV}$ never run at decode time — they live inside $\tilde W^{Q}$ and $\tilde W^{O}$.
- The only things read from HBM per step are the cached $c^{KV}$ (512) and $k^R$ (64) — 576 scalars/token — which is exactly the memory-bandwidth bottleneck MLA was designed to shrink.
- Same numerical result as the prefill-form trace above; just a different (associativity-exploiting) factorization of the matmuls.

## DeepSeek-V3: Scaling & Hardware Optimization

### FP8 Mixed-Precision Training
The first validated FP8 training of a frontier-scale (671B) model, tuned for H800 clusters.

- **All tensors in E4M3 — not the usual hybrid.** Prior FP8 recipes (NVIDIA Transformer Engine) used a *hybrid* format: E4M3 for the forward pass, E5M2 for gradients (Dgrad/Wgrad), buying extra exponent range for gradients. **V3 instead uses E4M3 everywhere**, accepting its narrower dynamic range because fine-grained scaling compensates. (A common misconception is "E4M3 forward / E5M2 backward" — that's the *old* approach V3 deliberately moved away from.)
- **Fine-grained quantization is what makes E4M3-everywhere work.** Instead of one scale per tensor, V3 scales small groups: **activations tile-wise (1×128)** and **weights block-wise (128×128)**. Sharing an exponent across a small group mitigates FP8's limited range, so outliers in one group don't wreck precision for the rest.
- **Online (not delayed) scaling.** Scale factors come from the current tensor's actual max, not a running history of past maxima — more robust to sudden spikes.
- **Mixed precision elsewhere.** Optimizer states are kept in BF16, a few sensitive ops stay higher-precision, and FP8 GEMMs let activations be stored in FP8 for the backward pass — roughly halving memory and boosting throughput vs. BF16.

### Multi-Token Prediction (MTP)
A training objective where the model predicts several future tokens, creating a denser learning signal — and doubling as a speculative-decoding drafter at inference.

- **Sequential / cascaded modules — NOT parallel towers.** This is the defining detail. The main model predicts $t_{i+1}$ from the shared trunk. Then **MTP module 1** (a single lightweight Transformer layer) takes the trunk's hidden states *plus the embedding of the just-predicted token* to predict $t_{i+2}$; **MTP module 2** chains off module 1 to predict $t_{i+3}$; and so on. Each module **depends on the previous one**, preserving the full causal chain. (Gloeckle et al.'s original MTP uses *parallel independent heads* off one trunk — V3 deliberately chose the sequential version to model inter-token dependencies.)
- **Shared embedding & output head.** Every MTP module reuses the main model's embedding and unembedding/output head — only the small per-module Transformer layer is extra. Open-source V3 ships **1 MTP module** (depth = 1).
- **Auxiliary training loss.** Each module's future-token prediction carries its own cross-entropy loss, added (weighted) to the main loss. Penalizing bad "look-ahead" guesses forces the trunk to build representations useful for longer-range planning — helping coherence in reasoning, code, and math.
- **Inference: default single-token, optional speculative decoding.** By default only the main head runs (max accuracy). But the MTP modules can act as a built-in **draft model**: they propose the next few tokens, and the main model verifies them in parallel in one forward pass. V3's MTP-1 has an **~80–90% acceptance rate**, giving **~1.8× higher generation throughput**.

### Auxiliary-Loss-Free Load Balancing
A MoE routing fix that keeps experts evenly used **without** a balancing loss.

- **The problem:** classic MoE adds an auxiliary balance loss to stop a few experts hogging tokens — but that loss competes with the language objective and can hurt quality.
- **V3's fix:** add a **per-expert learnable bias** to the routing affinity scores, used **only for top-k selection** (not for the gating weights in the weighted sum). The bias is nudged up for under-used experts and down for over-used ones, equalizing load. Because it only steers selection and isn't a gradient loss term, it balances experts **without distorting the training signal**. (A tiny sequence-level balance loss is kept only as a safety net.)

### DualPipe

Introduced by DeepSeek, it's a **bidirectional pipeline-parallelism** schedule that nearly eliminates pipeline bubbles by overlapping forward/backward computation with the expensive cross-node MoE communication.

**Background — why pipeline parallelism has "bubbles."** Pipeline parallelism splits a model's layers across GPUs (stage 0 = first layers, stage 1 = next, …). A batch is broken into micro-batches that flow stage→stage. But at the start (fill) and end (drain) of each step, some stages sit idle waiting for others — these idle gaps are **bubbles**. Standard schedules like **1F1B** (one-forward-one-backward) reduce but don't remove them, and the bubble fraction grows with the number of stages.

**The extra problem in MoE.** DeepSeek-V3's experts are spread across nodes (expert parallelism), so each layer needs **all-to-all dispatch** (send tokens to their experts) and **all-to-all combine** (gather results) over the inter-node network — expensive communication that, if left exposed, stalls compute.

**DualPipe's approach — feed from both ends + overlap everything.**
- **Bidirectional micro-batch flow.** Instead of micro-batches entering only at stage 0, DualPipe injects them from **both ends of the pipeline simultaneously** — one stream front→back, one back→front. The two streams interleave so stages stay busy from both directions, filling the bubbles each direction would leave.
- **Fine-grained overlap of compute and communication.** Each micro-batch's work is decomposed into components — **attention, all-to-all dispatch, MLP, all-to-all combine** — and the schedule is arranged so that while one micro-batch is *computing* (attention/MLP), another's *communication* (dispatch/combine) runs concurrently on the same GPU. The communication is hidden behind computation.
- **Backward split (Zero-Bubble-style).** The backward pass is divided into "gradient w.r.t. input" (B) and "gradient w.r.t. weights" (W), which can be scheduled independently to fill more gaps.

**The cost.** Bidirectional flow means DualPipe must keep **two copies of the model parameters** in memory (one per direction), plus higher activation memory. That's the price for near-zero bubbles. DeepSeek accepted this because the overlap let them train **without expensive tensor parallelism**, a net win at their scale.

**Net effect.** Computation and communication are almost fully overlapped, bubbles shrink to nearly zero, and cross-node MoE training becomes efficient enough to scale the model without extra compute overhead.

### Reasoning (Chain-of-Thought) via Distillation from R1
DeepSeek-V3 the **chat model** acquires step-by-step reasoning (long chain-of-thought, self-verification) by **distilling it from DeepSeek-R1**, the RL-trained reasoning model — V3 is not itself the RL-reasoning model.

**The pipeline (keep these three players straight):**
1. **DeepSeek-V3-Base** — the pre-trained foundation model (dense knowledge, no special reasoning behavior).
2. **DeepSeek-R1** — a *reasoning* model trained on top of V3-Base with **large-scale Reinforcement Learning** (using **GRPO**, Group Relative Policy Optimization). RL is what teaches it to produce long CoT, verify its own steps, and reason in a `<think>` block before answering.
3. **DeepSeek-V3 (chat)** — during post-training, reasoning ability is **distilled from R1 into V3**: R1 generates high-quality reasoning traces, and V3 is fine-tuned to imitate them. So V3 inherits *the behavior* without going through the full RL process itself.

**Why distill instead of just using R1?** Distillation transfers R1's reasoning competence into the general-purpose V3 while keeping V3's broad capabilities, output style, and controllable length — you get reasoning gains on math/code/logic without making the model a pure reasoning specialist.

**Caveat — don't conflate V3 and R1.**
- **R1 = the reasoning model** (RL-trained, the source of the `<think>`/CoT behavior).
- **V3 = the general foundation/chat model** that *benefits from R1's distillation*.
- The RL pipeline, GRPO, and the reasoning behavior properly belong to **R1**; V3 is the recipient. Attributing "trained with RL to reason" to V3 itself is the common mistake.

## DeepSeek-V4

Two model sizes: **V4-Pro** (1.6T total / 49B active) and **V4-Flash** (284B / 13B active), both natively supporting **1M-token** context. Headline efficiency: at 1M context, V4-Pro uses ~27% of V3.2's single-token FLOPs and ~10% of its KV cache; V4-Flash ~10% FLOPs and ~7% KV cache.* Carried over from V3 (unchanged or near-unchanged):
- **DeepSeekMoE** FFN (fine-grained routed experts + shared experts)).
- **Multi-Token Prediction (MTP)** — identical strategy to V3 (MTP depth = 1 in both V4 models).
- **Transformer backbone, tokenizer (128K vocab), token-splitting, Fill-in-Middle** — inherited from V3.

### 1. MoE changes (DeepSeekMoE)

- **Affinity activation changed:** the router's affinity-score function goes from **Sigmoid(·)** (V3) → **Sqrt(Softplus(·))** (V4).
  - The issue with Sigmoid is it saturates — once a logit is large, Sigmoid flattens to ~1, so the gradient vanishes and the router stops being able to sharpen or distinguish among strong experts. Softplus (a smooth ReLU, log(1+eˣ)) doesn't saturate at the top — it keeps growing, so strong affinities stay differentiable and the router can keep learning fine distinctions. The outer Sqrt compresses the dynamic range (tames very large values) so the scores don't blow up and stay numerically stable.  
- **Load balancing:** keeps the **auxiliary-loss-free** strategy, now augmented with a *slight sequence-wise balance loss* to prevent extreme imbalance within a single sequence.
  - It almost certainly is (this is the standard DeepSeek sequence-wise balance loss, carried from V2/V3's formulation): for each sequence, with `N` routed experts and top-`K` routing over `T` tokens:
    
  $$\mathcal{L}_{bal} = \alpha \sum_{i=1}^{N} f_i \, P_i$$

  - where:
    - $f_i$ = the fraction of the sequence's tokens routed to expert $i$ (scaled by $\frac{N}{K \cdot T}$),
    - $P_i$ = the average routing probability assigned to expert $i$ across the sequence,
    - $\alpha$ = the balance loss weight (set to **0.0001** in V4).
    - Minimizing the product $f_i \, P_i$ penalizes experts that receive both a high token-share **and** a high routing probability, pushing load to spread evenly across experts. 
- **Routing node constraint removed:** V4 drops V3's cap on the number of routing target nodes, with a redesigned parallelism strategy to keep training efficient.\
  - In V3, expert parallelism spreads experts across many nodes (machines). To limit cross-node communication, V3 used node-limited (device-limited) routing: each token's top-K experts were restricted to lie within at most M nodes (V3 used M=4) — the router first picked the few best nodes, then chose experts only within them. This caps how many machines a token's tokens must be dispatched to. V4 removes that constraint (a token can route to experts on any node) and redesigns the parallelism strategy to keep communication efficient:
    - MegaMoE fused kernel — the single fused dispatch→GMM1→SwiGLU→GMM2→combine kernel that implements overlap of microbratches of current tokens with previous.
    - Pull-based dispatch — each GPU reads activations from remote GPUs (lower notification latency than push).   
- **Hash routing in early layers:** the dense FFN layers in the first few Transformer blocks (V3) are replaced with **MoE layers using Hash routing** (first 3 MoE layers), where target experts are picked by a fixed hash of the token ID.
  - Hash routing is fixed and non-learned: the target expert(s) for a token are determined by a predefined hash function of the token ID — i.e. token "the" always goes to the same expert, decided by hash(token_id), regardless of context or training. Why use it in the first few layers: early layers do mostly low-level, token-identity-driven processing, so a deterministic hash is a cheap mechanism, and it sidesteps the routing-induced outliers the report blames for early-training loss spikes. V4 uses it for the first 3 MoE layers.  
  - The cite "Hash Layers for Large Sparse Models" (Roller et al., 2021) as hashing mechanism: in that work, the hash is a fixed precomputed lookup table mapping each token ID → expert (built once **before training**), most simply by assigning token IDs to experts via a balanced/random partition (e.g. modulo or frequency-balanced bucketing) so the table spreads tokens evenly. At runtime, they use **a frequency-balanced lookup at O(1) to look for the hash of each index**. How's that frequency table built (one common way):
    1. Count each token's frequency over a corpus.
    2. Sort tokens by frequency (most common first).
    3. Greedily assign tokens to experts to equalize total load — e.g. a bin-packing / "least-loaded bin" pass: give each next token to whichever expert currently has the lowest accumulated frequency.
    4. Freeze the result as a lookup table token_id → expert. 

During **post-training**:
- **FP4 for routed experts:** routed expert parameters use **FP4** precision (FP4×FP8); same peak FLOPs as FP8×FP8 on current hardware but ~1/3 more efficient on future hardware.
- **FP4 Quantization-Aware Training (post-training):** applied to MoE expert weights and the indexer QK path to cut memory/compute.
- **Two-stage post-training pipeline:** (1) train independent domain **specialist** experts (SFT + RL via **GRPO**); (2) consolidate them into one unified model via **on-policy distillation** (student minimizes reverse-KL to teachers).
 
### 2. Manifold-Constrained Hyper-Connections (mHC) - replaces conventional residual connections to stabilize very deep training 

Replaces  conventional residual connections ($X_{l+1} = X_l + F_l(X_l)$, where $F_l$ is the layer and $X_l \in \mathbb{R}^d$ is the hidden state). **Hyper-Connections (HC) widens the residual stream by a factor $n_{hc}$**, so the residual state becomes $X_l \in \mathbb{R}^{n_{hc} \times d}$ (i.e. $n_{hc}$ parallel copies of the hidden stream instead of one). It then learns three small mappings that mix these copies:
- an **input mapping** $A_l \in \mathbb{R}^{1 \times n_{hc}}$ — picks which combination of the $n_{hc}$ streams feeds into the layer,
- a **residual-mixing matrix** $B_l \in \mathbb{R}^{n_{hc} \times n_{hc}}$ — mixes the streams among themselves,
- an **output mapping** $C_l \in \mathbb{R}^{n_{hc} \times 1}$ — distributes the layer's output back across the streams.

The update becomes:

$$X_{l+1} = B_l X_l + C_l \, F_l(A_l X_l)$$

**The problem:** stacked over many layers, the repeated multiplication by $B_l$ makes the signal grow or shrink uncontrollably, causing numerical instability. **mHC's fix** forces each $B_l$ to be a **doubly stochastic matrix** (every row and column sums to 1, all entries $\geq 0$) — the set known as the **Birkhoff polytope** $\mathcal{M}$:

$$B_l \in \mathcal{M} := \{\\, M \in \mathbb{R}^{n \times n} \mid M \mathbf{1}_n = \mathbf{1}_n, \\; \mathbf{1}_n^T M = \mathbf{1}_n^T,\\; M \geq 0 \\,\}$$

This bounds its spectral norm $\lVert B_l \rVert_2 \leq 1$, so the residual transform is **non-expansive** (it can't amplify the signal). And because $\mathcal{M}$ is **closed under multiplication**, stacking many such layers stays stable in both the forward and backward passes.

### 3. Hybrid Attention: CSA + HCA — replaces V3's MLA

Two attention types are **interleaved across layers**. Both compress the KV cache by **learned softmax-weighted pooling of a block of tokens into a single entry** (a `Z` projection produces pooling weights, plus learnable positional biases), then run **MQA** over a single shared compressed KV head.

### Hybrid Attention: CSA + HCA 

DeepSeek-V4 replaces V3's MLA with two compressed attention types, **interleaved across layers**. Both shrink the KV cache by **pooling a block of consecutive tokens into a single compressed entry** (a learned, softmax-weighted average), then run attention over those few entries. They differ in *how hard* they compress and *whether* they then sparsify.

First, the shared compression idea. For input hidden states $H \in \mathbb{R}^{n \times d}$ ($n$ = sequence length, $d$ = hidden size), the model produces per-token KV vectors $C = H W^{KV}$ and matching **compression weights** $Z = H W^{Z}$. To compress a block of tokens into one entry, it softmaxes the block's $Z$ values into pooling weights $S$ and takes the weighted sum of that block's $C$ vectors:

$$S = \mathrm{Softmax_{row}}(Z + B), \qquad C^{Comp}_i = \sum_{j \in \text{block } i} S_j \odot C_j$$

where $B$ is a learnable positional bias and $\odot$ is element-wise product. So "compression" = a **learned weighted pooling** of every block of tokens into one KV entry — not simple averaging, and not MLA-style per-token shrinking.

#### Compressed Sparse Attention (CSA)

- **Compress (mild):** every $m$ tokens → 1 entry, so the sequence shrinks to $\tfrac{1}{m}$ ($m = 4$). CSA uses **two parallel streams** $a$ and $b$ whose blocks are offset, so consecutive compressed entries **overlap** in which tokens they cover — this avoids hard block boundaries losing information.
- **Sparse select (DeepSeek Sparse Attention, via a "Lightning Indexer"):** after compression there are still many entries, so each query attends to only the most relevant ones. A lightweight low-rank **indexer** scores query $t$ against each compressed block $s$:

  $$I_{t,s} = \sum_{h=1}^{n^I_h} w^I_{t,h} \cdot \mathrm{ReLU}\!\left(q^I_{t,h} \cdot K^{IComp}_s\right)$$

  where $q^I$ are indexer query heads, $K^{IComp}$ are compressed indexer keys, and $w^I$ are per-head weights. A **top-$k$** selector then keeps only the $k$ highest-scoring compressed entries ($k = 512$ for Flash, $1024$ for Pro):

  $$\mathcal{C}^{SprsComp}_t = \left\{\, C^{Comp}_s \;\middle|\; I_{t,s} \in \mathrm{Top\text{-}k}(I_{t,:}) \,\right\}$$

- **Core attention:** standard attention (MQA, below) over just those $k$ selected entries. The query is built low-rank from a compressed latent $c^Q_t = h_t W^{DQ}$, which is **shared** with the indexer's query — so the indexer adds little extra cost.

#### Heavily Compressed Attention (HCA)

- **Compress (aggressive):** uses the **same pooling mechanism** but a much larger rate $m' \gg m$ ($m' = 128$), and a **single non-overlapping stream**. The sequence shrinks to $\tfrac{1}{m'}$ — a far smaller cache.
- **Dense attention:** because there are now so few entries, HCA **skips the indexer and top-$k$ entirely** — each query attends to **all** the compressed entries. No sparsity overhead, no approximation, just a tiny dense cache.

The intuition for pairing them: CSA keeps a **richer, selectively-accessed** cache (good for reaching specific distant tokens), while HCA keeps a **tiny, fully-read** cache (cheap, good where coarse context suffices). Interleaving the two across layers balances fidelity against cost.

#### Details shared by both CSA and HCA

- **Shared-KV MQA + grouped output projection.** Core attention is **Multi-Query Attention**: a single compressed KV head serves as both key and value for all query heads. Because the concatenated head outputs ($c \cdot n_h$ wide) are large, the output is projected back to $d$ in two cheaper steps — split the $n_h$ head outputs into $g$ groups, project each group to a smaller intermediate dim $d_g$, then project the concatenation to the final $\hat{o}_t \in \mathbb{R}^d$.
- **Query/KV RMSNorm.** An RMSNorm is applied to queries and the compressed KV entry just before core attention, preventing attention logits from exploding (and removing the need for QK-Clip in the Muon optimizer).
- **Partial RoPE.** Rotary position embedding is applied to only the **last 64 dimensions** of queries and KV entries. Since each entry is used as both key *and* value, the naive output would carry absolute positions; as a fix, RoPE with position $-i$ is also applied to each output $o_{t,i}$, so the result carries **relative** position (each entry's contribution depends on its distance from the query).
- **Sliding-window branch.** Under causal masking, a query can't see the recent tokens inside its own not-yet-compressed block — but recent tokens matter most for language modeling. So each query additionally attends to the $n_{win} = 128$ most recent **uncompressed** tokens, concatenated with the compressed entries.
- **Attention sink.** Each head has a learnable sink logit $z'_h$ added to the softmax **denominator**:

  $$s_{h,i,j} = \frac{\exp(z_{h,i,j})}{\sum_k \exp(z_{h,i,k}) + \exp(z'_h)}$$

  This lets a head's attention weights sum to **less than 1** (even near 0) — i.e. a head can choose to "attend to nothing" when appropriate.
- **Mixed-precision KV storage.** The RoPE dimensions are stored in **BF16**, the rest in **FP8** — roughly halving the cache vs. pure BF16. The lightning indexer itself computes in **FP4**.
- **Net result.** Taking BF16 GQA-8 (head dim 128) as a baseline, DeepSeek-V4's KV cache shrinks to about **2%** of it in the 1M-token setting.

### Muon Optimizer - replaces AdamW for faster convergence and better training stability

**Background:** a weight matrix's gradient (and its momentum) tends to be dominated by a few directions — a handful of large singular values, with the rest small, which slows learning and can destabilize training. Muon's idea is to **orthogonalize the update**: rescale all those directions to comparable magnitude so the step explores every direction of the weight matrix evenly.

**Core mechanism.** For a weight matrix's momentum update $M$ with singular value decomposition $M = U \Sigma V^T$, the ideal orthogonalized update is $U V^T$ — i.e. keep the *directions* ($U$, $V$) but flatten all the singular values in $\Sigma$ to 1. Computing a real SVD every step would be far too expensive, so Muon approximates $U V^T$ with cheap matrix multiplications via **Newton-Schulz iterations**. Each iteration is a fixed polynomial in $M$:

$$M_k = a\,M_{k-1} + b\,(M_{k-1} M_{k-1}^T)\,M_{k-1} + c\,(M_{k-1} M_{k-1}^T)^2 M_{k-1}$$

Repeating this drives every singular value of $M$ toward 1 (leaving the orthogonal factors $U V^T$), without ever forming an explicit SVD.

The full step is then: accumulate (Nesterov) momentum → orthogonalize via Newton-Schulz → **rescale the update's RMS** by $\sqrt{\max(n,m)}\cdot\gamma$ (so AdamW's learning-rate hyper-parameters can be reused unchanged) → apply weight decay and update.

**Hybrid Newton-Schulz (V4's tweak).** V4 runs **10 iterations in two stages** with different coefficients $(a,b,c)$:
- **First 8 iterations** use aggressive coefficients $(3.4445,\,-4.7750,\,2.0315)$ to converge fast — quickly pulling the singular values near 1.
- **Last 2 iterations** switch to gentle coefficients $(2,\,-1.5,\,0.5)$ to settle the singular values **precisely at 1** without overshooting.

The two-stage split gets both speed (aggressive early) and accuracy (stable finish) — a single coefficient set can't do both well.

**What stays on AdamW.** Muon only makes sense for 2D weight *matrices* (where orthogonalization is meaningful). So V4 keeps **AdamW** for the embedding module, the prediction head, the static biases and gating factors of mHC, and all RMSNorm weights — and uses **Muon** for everything else.

**No QK-Clip needed.** Some Muon recipes add "QK-Clip" to stop attention logits from exploding. V4 doesn't need it: because RMSNorm is applied directly to the queries and KV entries (see the attention section), the logits are already bounded, so the trick is dropped.

### 5. Infrastructure — training & inference

#### 5.1 Fine-grained EP communication-computation overlap (single fused MegaMoE, a CUDA mega-kernel in DeepGEMM)

- **Insight:** within an MoE layer the 4 stages are 2 comm-bound (Dispatch, Combine) + 2 compute-bound (Linear-1+SwiGlu, Linear-2); profiling shows total comm < total compute, so comm can be fully hidden behind compute.
- **Wave-based fine-grained pipeline:** experts are split into **waves**; as soon as a wave's communication finishes, its computation starts without waiting for other experts. In steady state, current-wave compute, next-wave token transfer, and completed-expert result-sending all run concurrently.
- **MegaMoe vs. Comet:** The chain is Dispatch → Linear-1 (+SwiGLU) → Linear-2 → Combine.
  - Comet: overlaps Dispatch with Linear-1, and Linear-2 with Combine, separately. So it's two overlap regions: {Dispatch+Linear-1} and {Linear-2+Combine}. Each region overlaps one communication phase with one matmul. Theoretical speedup the paper attributes to it: 1.42×.
    - It's also **reordering the order in which the GEMM computes its output tiles so it matches the order in which token tiles arrive over the network**. A normal GEMM walks its tiles in a fixed, communication-blind order (tile 0, 1, 2…). But dispatch delivers token tiles in some other order, so a fixed traversal would stall on tile 0 while tiles 5–9's data is already sitting there unused. Comet's dependency-resolving step computes which token tile each GEMM tile needs, then reorders the GEMM's tile traversal to consume tiles in arrival order  
  - MegaMoE: achieves finer-grained overlapping by splitting and scheduling experts into waves — the whole chain in one kernel, with experts pipelined as waves so current-wave compute, next-wave dispatch, and prior-wave combine all run concurrently. Theoretical speedup: 1.92×.
  - How? Imagine Dispatch → matmul+swiglu (C=A@B) → matmul (E=C@D)→ Combine . We use expert parallelism fo B and D are stored fully on the expert's GPU. A is split row-size, so C is computed row-wise, so E is computed row-wise, and can be sent (point-to-point) to receiver of token to do a partial update.
  - Input A and expert weights B and D are fully resident in global memory. Intermediatte C stays on-chip / in registers if fused. BlockMoe fuses it, Commet doesn't, so C is materialized in global memory. Output E stays in global memory or on-chip if fused with combine.
  - The one real storage difference is C: because MegaMoE is a single kernel spanning GMM1→GMM2, C never leaves the SMs (produced as a tile, consumed by GMM2 immediately, discarded). Comet's two regions are separated by a kernel boundary, so C is materialized in HBM and re-read.
  - MegaMoE uses *expert wave pipelining*: Each wave is a subset of the local experts (e.g. experts 10–20, then 20–30, then 30–40…), and while one wave computes, the next wave's tokens are being dispatched in and the previous wave's results combined out — so compute and communication overlap across waves. If you wave by *experts*, each wave loads a few experts' weights to perform big GEMM — good arithmetic intensity. If you waved by *tokens*, a single token subset touches *many different experts* (each token's top-K span the whole expert pool), so you'd reload lots of expert weights per wave with tiny per-expert batches → poor GEMM efficiency and more weight traffic. Se we get bigger per-expert GEMMs (higher arithmetic intensity), and possibly fewer weight loads. Better than traditional MoE pipelining that sends waves by token groups. 
  - Comet notes:
    - To enable early computation by the experts, tokens are sorted based on their source rank
    - The shared tensor undergoes a top-k reduction after processing by the GroupGEMM of experts. 
    - The tile computation sequence is adjusted  to enable the consumer operator to start processing before expert computations are fully completed.
    - The compute sequence of tiles in the GroupGEMM is then designed to minimize dependency on remote data. 
    - Computation begins from tiles containing local tokens while the transfer of other remote tokens proceeds concurrently.
    - Instead of computing each expert sequentially, GroupGEMM operations are executed column-wise.  This allows the reduction and communicate operations to proceed as soon as the first $T_N$ columns of the shared tensors are computed.
    - My notes, in practice: take matmul+swiglu (C=A@B) → matmul (E=C@D)→ Combine.
      - In regular matmul, "for every row of A, for every column of B", where D is populated row-wise, if a token e.g. X would be in position 0 in rank 0 and position 9999 in rank 1, then it could only by reduced after the iteration that hits row 9999 in rank 1.
      - In comet, "for every column of B, for every row of A"(?), D is populated column-wise, therefore we get a subset of all columns of all tokens as we loop over D, allowing is to do partial reductions of tokens combine. Note the reduced token index across experts in Figure 6. 
    
Using **A** = dispatched tokens, **B** = weight1 (gate/up), **C** = swiglu(A·B), **D** = weight2 (down), **E** = C·D, **Y** = combined output.

| Tensor | Storage | Granularity | Comet vs MegaMoE |
|---|---|---|---|
| **A** (received tokens) | HBM (per-expert input buffer) | tiled into SM for the GEMM | same in both |
| **B** (weight1) | HBM, **whole per expert** (EP never splits weights) | streamed into SM tiles | same in both |
| **C** (intermediate) | **the key difference** | produced/consumed tile-by-tile | **Comet: HBM** (between the two regions); **MegaMoE: on-chip**, only a token-tile alive at a time, never written to HBM |
| **D** (weight2) | HBM, whole per expert | streamed into SM tiles | same in both |
| **E** (expert output) | HBM | tiled | same; MegaMoE may keep the E-tile on-chip into combine |
| **Y** (combined output) | HBM | per-token weighted sum | same in both |

The one real storage difference is **C**: because MegaMoE is a single kernel spanning GMM1→GMM2, C never leaves the SMs (produced as a tile, consumed by GMM2 immediately, discarded). Comet's two regions are separated by a kernel boundary, so C is materialized in HBM and re-read.

#### 5.2 TileLang DSL for kernels

- Replaces hundreds of fine-grained Torch ATen ops with fused kernels; balances productivity and runtime performance.
- **Host Codegen:** moves host-side checks into generated host code (TVM-FFI), cutting per-invocation CPU overhead from tens/hundreds of µs to **<1 µs**.
- **SMT-solver formal analysis:** integrates the **Z3** solver (QF_NIA) for integer/index reasoning, unlocking vectorization, barrier insertion, and simplification passes.
- **Precision control:** fast-math off by default; opt-in approximate ops and IEEE-754 intrinsics; aligns lowering with NVCC for **bitwise reproducibility**.

#### 5.3 Batch-invariant & deterministic kernels

- **Batch invariance** (a token's output is bitwise-identical regardless of batch position):
  - *Attention:* avoids split-KV; uses a **dual-kernel** decode (one SM per sequence for full waves + a multi-SM kernel with matched accumulation order for the tail) to dodge wave-quantization.
  - *Matmul:* replaces cuBLAS with **DeepGEMM**, abandons split-k (with optimizations to recover its performance).
- **Determinism** (removing atomic-add non-determinism in the backward pass): per-SM accumulation buffers + global deterministic sum for **attention backward**; token-order pre-processing + buffer isolation for **MoE backward**; split-out + deterministic reduction for the small (output-dim 24) **mHC matmul**.

#### 5.4 Training framework deltas

- **Hybrid ZeRO for Muon:** Muon needs the *full* gradient matrix, which conflicts with element-wise ZeRO. Dense params use a knapsack-balanced bucket assignment with a ZeRO-parallelism cap (redundant Muon compute beyond it); MoE params are flattened per-expert (down/up/gate) and evenly distributed. Newton-Schulz batched across same-shape params; MoE gradients quantized to **BF16 with stochastic rounding** (halving comm), reduced via all-to-all + local FP32 sum.
- **Cost-effective mHC:** fused mHC kernels, selective **recomputation** of inter-layer hidden states / normalized inputs, and an adjusted **DualPipe 1F1B** overlap — holding mHC's overhead to **6.7%** of the overlapped pipeline stage.
- **Two-stage contextual parallelism:** handles compressed attention across CP ranks — stage 1 sends each rank's last `m` uncompressed KV to the next rank (to fix block straddling); stage 2 all-gathers locally-compressed entries and reorganizes them with a fused select-and-pad operator.
- **Tensor-level activation checkpointing:** extends autograd (via TorchFX) so developers annotate *individual tensors* for recomputation; builds minimal recomputation subgraphs, frees/reuses storage with no GPU memcpy, and auto-dedups shared-storage tensors.

#### 5.5 Inference framework deltas (KV cache)

- **Heterogeneous KV cache layout:** hybrid attention creates multiple KV types (CSA main + CSA indexer + HCA + SWA + uncompressed tail), each with different sizes/update rules — breaking PagedAttention assumptions.
- **State cache:** SWA KV + not-yet-compressible tail tokens are treated as an SSM-like per-sequence state in a fixed-size pool.
- **Sparse-attention kernel co-design:** blocks sized to multiples of `lcm(m, m')` so variable tokens-per-block work without perf loss.
- **On-disk KV storage for shared-prefix reuse:** compressed CSA/HCA entries are stored to disk and reused on prefix hits (tail incomplete block recomputed). For the bulkier uncompressed SWA KV (~8× larger), three strategies trade storage vs. recompute: **Full SWA caching**, **Periodic checkpointing** (every `p` tokens), **Zero SWA caching** (recompute last `n_win·L` tokens).

### 6. Pre-training & stability

- **Data:** >32T tokens (Flash 32T, Pro 33T), with stronger long-document curation, multilingual expansion, agentic mid-training data, and web filtering against model collapse; **sample-level attention masking** (vs. V3).
- **Curriculum:** sequence length grows 4K → 16K → 64K → 1M; attention starts **dense** for the first ~1T tokens, then sparsity is introduced at 64K (with a lightning-indexer warmup stage).
- **Stability fixes for trillion-param MoE:** **(1) Anticipatory Routing** — decouple backbone vs. routing updates: compute features with current params `θ_t` but routing indices with historical `θ_{t−Δt}` (pre-computed/cached), triggered automatically on loss spikes (~20% local overhead, near-zero amortized). and  **(2) SwiGLU Clamping** — clamp the linear component to `[−10, 10]` and cap the gate at 10 to eliminate outliers.
